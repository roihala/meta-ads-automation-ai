## 1. The problem

When a scan finishes with zero proposals, the operator-facing `/runs` page makes it look like the agent did nothing. In reality the agent often:

- Pulled insights and observed N findings
- Was blocked from acting on M of them because a **capability** was unavailable (`tracking_verified`, `primary_kpi_set`, …)
- Had K proposals **rejected by guardrails** before they reached the queue
- Deliberately **skipped** campaigns (in learning phase, no spend, etc.)

Every one of these is already logged to `agent_decisions` with a `rationale`. The current detail page (`web/src/app/runs/[run_id]/page.tsx`, 319 lines) renders the trail but doesn't tell the **story** of why the run looked the way it did.

A debug-mode view should answer the operator's real question: *"what did the agent consider, what did it decide, and why?"*

## 2. Decisions

| # | Decision | Reason |
|---|---|---|
| 1 | **One route, not two.** Fold the narrative into the existing `/runs` and `/runs/[run_id]`. No `/debug/runs`. | Avoids parallel routes drifting apart. Operator + debug share the same data; debug is just deeper sections. |
| 2 | **Gate by `DEBUG=true` env var, server-side.** | No DB changes, no role system. Server reads `process.env.DEBUG === "true"` and passes a `debug` boolean prop into `Nav` and into the page components. Plain `DEBUG` (not `NEXT_PUBLIC_DEBUG`) works because the check happens in server components; `nav.tsx` is `"use client"` but it's invoked by a server caller that can pass props. |
| 3 | **No new tables, no new migrations.** | All data exists: `agent_decisions` (with `inputs`/`outputs`/`rationale`/`guardrail_violations`/`confidence`/LLM metadata), `approvals`, `heartbeats`. Migration 033 added `observation_blocked` + `finding_key`. |
| 4 | **Lean toward "key decisions and their explanations."** | The narrative sections are the headline feature. Raw trail + token telemetry are escape-hatch / supporting. |

## 3. What's debug-only vs always shown

### `/runs` (list)
- **Always shown:** the existing per-run row with counts (decisions, proposals, skips, rejections, errors, observation_blocked, campaigns_touched).
- **Debug-only additions:**
  - **Shape badge column:** `empty` / `blocked` / `rejected` / `productive` / `errored`. Derived from the counts in `RunSummaryRow`, no new query needed.
  - **Shape filter** in the query string.
  - **Token-burn column** (sum of input + output tokens across decisions for the run).

### `/runs/[run_id]` (detail)
- **Always shown:** the existing layout (whatever it currently renders — header, gate chips, decision trail).
- **Debug-only additions:** the five new sections in §5. They render below the current operator surface, behind an `if (debug)` server-side conditional, so non-debug builds never ship the markup.

## 4. Data — what we query, no new tables

Reuse the existing adapter methods:

| Method | File | Used for |
|---|---|---|
| `listRunsForBusiness(businessId, opts)` | `web/src/lib/db/local-postgres.ts:775` | List page (already used) |
| `listDecisionsForRun(businessId, runId)` | `web/src/lib/db/local-postgres.ts` | Detail page (already used) |

The new sections all derive from the existing decision rows. **No new SQL** is required for v1 — everything is computed in TypeScript from the array of `AgentDecision`.

If pivoting to SQL aggregates later (when row counts grow), the natural breakouts are:

- Phase timings → `SELECT node_name, MIN(created_at), MAX(created_at) FROM agent_decisions WHERE run_id = $1 GROUP BY node_name`
- Guardrail histogram → `SELECT unnest(guardrail_violations) AS code, COUNT(*) FROM agent_decisions WHERE run_id = $1 GROUP BY code`
- Token totals → `SELECT SUM((outputs->'llm'->>'input_tokens')::int), SUM((outputs->'llm'->>'output_tokens')::int) FROM agent_decisions WHERE run_id = $1`

For v1, do them client-side on the already-fetched array. Add SQL when slow.

## 5. The five debug sections on the detail page

In render order, **below** the existing operator content:

### 5.1 TL;DR card — "Why this run looked the way it did"

A single auto-generated paragraph derived from the counts:

> *"The agent observed 8 issues across 3 campaigns. It would have proposed 5 changes, but 4 were blocked because the Pixel is not yet verified, and 1 was rejected by the budget-cap guardrail. 0 proposals reached the queue."*

Plain Hebrew, generated by a small pure function `buildRunNarrative(decisions: AgentDecision[]): string`. Lives in `web/src/lib/runs-summary.ts` next to the existing summary helper.

### 5.2 Capability snapshot at run start

The 13 capabilities (`scale_up`, `new_creative`, `set_kpi_target`, `ab_test_decide`, …) as chips:

- **Available** chips: green-ish
- **Blocked** chips: muted, with a tooltip / inline list of `blocked_by` requirements (`tracking_verified`, `primary_kpi_set`, etc.)

Source: the boot-phase `observation` row's `outputs.capabilities` — written by `runners/daily_observe_propose.sh` after it calls `compute_capabilities`. If the row isn't present (older runs), section renders "Capability snapshot not recorded for this run."

### 5.3 Key decisions, grouped

Four accordions, in this order:

1. **🛑 Blocked findings** (`decision_type = 'observation_blocked'`)  
   Each item: finding type · target (campaign/adset) · `outputs.would_propose` summary · `outputs.blocked_by` requirement chips · full `rationale`. This is the headline of the page when nothing got proposed.

2. **❌ Guardrail rejections** (`decision_type = 'rejection'`)  
   Each item: what was being proposed (from `inputs`) · `guardrail_violations[]` as code chips · `rationale`. A small histogram at the top shows which codes fired most often.

3. **⏭️ Deliberate skips** (`decision_type = 'skip'`)  
   Campaign · reason from `rationale`. Compact list — usually short and not interesting individually.

4. **✅ Proposals created** (`decision_type = 'proposal'`)  
   Each item: task_type · summary · link to `/approvals/[related_approval_id]`. Already covered by operator view; here it's restated for completeness so the four-bucket sum equals total decisions.

5. **💥 Errors** (`decision_type = 'error'`)  
   If any. Stack/message, expandable.

### 5.4 Per-campaign lane

Vertical swim-lane: one column per campaign touched in the run, decisions ordered top-to-bottom by `created_at`. Each cell is a tiny `decision-row` variant.

Answers: *"what did the agent think specifically about campaign X this run?"*

### 5.5 Raw decision trail (collapsible) + token & latency telemetry

- **Raw trail:** the flat existing rendering, in a `<details>` so it's out of the way unless wanted.
- **Token / latency:** small table at the bottom — per `node_name`, sum of input tokens, output tokens, total latency_ms. Pulled from `outputs.llm` (model, input_tokens, output_tokens, latency_ms) recorded by `log_decision.py`. Useful for spotting expensive runs and prompt regressions.

## 6. Implementation plumbing

### 6.1 Visibility — the `DEBUG` flag

**Read it in a single place** so it's grep-able:

```ts
// web/src/lib/debug.ts (new, ~5 lines)
export function isDebugMode(): boolean {
  return process.env.DEBUG === "true";
}
```

Two call sites:

1. **Layout / page that renders `<Nav>`** — passes `debug={isDebugMode()}` as a prop. `Nav` accepts a new optional prop and renders the debug-only menu treatments (shape badges in `/runs`, "debug" badge on the nav item itself if we want a visual hint).
2. **`runs/[run_id]/page.tsx`** — reads `isDebugMode()` server-side and conditionally renders sections 5.1–5.5.

`DEBUG` is plain (not `NEXT_PUBLIC_*`) because every read is in a server component. If we ever need it client-side, switch to `NEXT_PUBLIC_DEBUG` — but for now, server-side is enough and keeps it out of the JS bundle.

## 7. Build order

1. `lib/debug.ts` + plumb `debug` prop through Nav and the two run pages (no UI yet).
2. `lib/runs-summary.ts` helpers + Vitest tests.
3. Section 5.1 (TL;DR) — fastest win, validates the data shape.
4. Section 5.3 (decision groups) — the headline feature.
5. Section 5.2 (capability snapshot) — requires that the boot observation actually carries `outputs.capabilities`; verify with a local scan first.
6. Section 5.4 (per-campaign lanes).
7. Section 5.5 (raw trail + telemetry).
8. `/runs` list — shape badge + filter.

Each step is a small PR; stop and review after #4 to confirm the narrative framing lands before building the rest.

## 8. Open questions / future

- **Does the boot observation reliably carry `outputs.capabilities`?** Verify on the next local run. If not, add it to the boot `log_decision` call in `CAMPAIGNER.md` instructions, or recompute via `compute_capabilities` at render time.
- **Run replay / "what would happen if capability X were unblocked?"** — out of scope for v1 but a natural follow-up.
- **Cross-run trends** (capability blockage over time, guardrail histogram drift) — belongs on a separate dashboard, not the per-run page.
- **Public exposure later:** if/when operators want this narrative without the `DEBUG` flag, drop the gate. The page works the same.
